# PLAN.md

# План MVP: AI-переводчик DOCX-документов на Python

## 1. Цель первой версии

Создать минимально рабочее приложение для автоматического перевода DOCX-документов с сохранением базовой структуры файла.

Поток MVP:

1. Пользователь загружает DOCX-файл.
2. Выбирает исходный и целевой язык.
3. Система извлекает текстовые блоки из документа.
4. Текст отправляется в AI-переводчик смысловыми блоками.
5. Переведенный текст вставляется обратно в документ.
6. Пользователь получает новый DOCX-файл.

Экономия токенов:

- В первой версии обрабатывать только DOCX, без PDF/XLSX/TXT.
- Не отправлять в AI пустые строки, пробелы, номера страниц и служебные элементы.
- Переводить только изменяемый текст, не отправляя структуру документа, стили, изображения и бинарные данные.

## 2. Границы MVP

Входит в MVP:

- Загрузка одного DOCX-файла.
- Выбор языка `FROM` и `TO`.
- Запрет одинаковых языков.
- Перевод абзацев и текста в таблицах.
- Сохранение исходного DOCX как шаблона и замена текста внутри него.
- Базовое сохранение форматирования за счет редактирования существующего документа.
- OpenAI-совместимый AI-клиент.
- Простой FastAPI endpoint.
- Сохранение результата в папку `outputs`.
- Логирование этапов обработки.

Не входит в MVP:

- PDF, XLSX, TXT.
- Пакетная загрузка файлов.
- Web UI.
- Drag & drop.
- Глоссарии и пользовательские словари.
- Точный расчет стоимости до запуска.
- Редактирование изображений, формул и диаграмм.
- Продвинутая очередь фоновых задач.

Экономия токенов:

- Не реализовывать несколько AI-движков сразу, оставить интерфейс так, чтобы позже добавить DeepL, Google Translate и Custom API.
- Не переводить подписи к изображениям отдельной логикой в MVP, если они не являются обычным текстовым абзацем.
- Не добавлять сложные промпты для каждого типа документа, использовать один короткий системный шаблон.

## 5. Поддерживаемые языки

Список языков MVP:

- Русский: `ru`
- Английский: `en`
- Латышский: `lv`

Правила:

- Если языки одинаковые, API возвращает ошибку `400`.
- Внутри приложения использовать коды языков, а человекочитаемые названия хранить отдельно.

Экономия токенов:

- Не отправлять список всех языков в каждый запрос к AI.

## 6. API MVP

### POST `/translate/`

Назначение:

- Принять DOCX-файл.
- Принять `source_lang`.
- Принять `target_lang`.
- Запустить перевод.
- Вернуть путь или ссылку на готовый файл.

Поля формы:

- `file`: DOCX-файл.
- `source_lang`: код исходного языка.
- `target_lang`: код целевого языка.

Ответ:

```json
{
  "status": "completed",
  "file_name": "originalname_translated_to_en.docx",
  "file_path": "outputs/originalname_translated_to_en.docx"
}
```

Ошибки:

- `400`: неверный формат файла.
- `400`: одинаковые языки.
- `413`: файл слишком большой.
- `502`: ошибка AI API.
- `500`: внутренняя ошибка обработки документа.

Экономия токенов:

- Не возвращать в API полный текст документа.
- Не сохранять полный prompt/response в обычные логи.
- В ошибках хранить краткое описание, а подробности включать только в debug-режиме.

## 7. Обработка DOCX

Основной принцип:

- Открывать исходный DOCX через `python-docx`.
- Идти по абзацам и таблицам.
- Извлекать текстовые блоки с привязкой к месту в документе.
- Переводить только текст.
- Заменять текст в существующем документе, чтобы сохранить структуру.

Обрабатываемые элементы MVP:

- Обычные абзацы.
- Заголовки, если они представлены как абзацы со стилями.
- Ячейки таблиц.
- Маркированные и нумерованные списки, если они представлены как абзацы.

Не изменять:

- Изображения.
- Формулы.
- Диаграммы.
- Пустые абзацы.
- Кодовые блоки, если они определены стилем или простым эвристическим правилом.

Экономия токенов:

- Не отправлять весь DOCX в AI.
- Не отправлять XML-разметку документа.
- Не переводить одинаковые повторяющиеся строки повторно, использовать кэш по нормализованному тексту.
- Пропускать строки короче заданного порога, если они похожи на технические обозначения: `ISO 9001`, `EN 123`, `GOST`, `v1.2.3`.

## 8. Сегментация текста

Задача сегментации:

- Объединять короткие текстовые блоки в пачки.
- Не превышать лимит символов на один AI-запрос.
- Сохранять соответствие между исходным блоком и переводом.

Базовый подход:

- Каждый абзац или ячейка таблицы получает `block_id`.
- Несколько блоков объединяются в batch.
- В AI отправляется JSON-массив объектов `{ "id": "...", "text": "..." }`.
- AI возвращает JSON-массив `{ "id": "...", "translation": "..." }`.
- Перевод вставляется обратно по `id`.

Рекомендуемые лимиты MVP:

- До 20-40 блоков в одном batch.
- До 6 000-10 000 символов исходного текста на один запрос.
- Отдельный retry для каждого batch.

Экономия токенов:

- Batch-запросы дешевле, чем отдельный запрос на каждый абзац.
- Не делать слишком большие batch-запросы, чтобы не терять весь пакет при ошибке.
- Использовать короткие ключи JSON только при необходимости, например `id` и `text`.
- Не включать в каждый batch длинные инструкции, держать prompt коротким и стабильным.

## 9. Prompt для перевода

Базовый системный смысл:

```text
Translate from {source_lang} to {target_lang}.
Preserve meaning, terminology, tone, numbers, tags, codes, and formatting markers.
Return only valid JSON with the same ids.
Do not translate code, formulas, standards names, or product identifiers.
```

Формат пользовательского сообщения:

```json
[
  {"id": "b1", "text": "Source text"},
  {"id": "b2", "text": "Another source text"}
]
```

Экономия токенов:

- Не просить AI объяснять перевод.
- Не просить AI возвращать исходный текст.
- Не добавлять примеры в prompt на каждом запросе.
- Не использовать длинные роли и описания, если короткая инструкция дает стабильный результат.

## 10. AI-клиент

Первая реализация:

- OpenAI-совместимый клиент.
- Модель задается через `.env`.
- API key хранится только в `.env`.
- Температура низкая: `0` или `0.1`.
- Повтор запроса при временных ошибках.
- Таймаут на запрос.

Настройки `.env`:

```text
OPENAI_API_KEY=
OPENAI_MODEL=gpt-4o-mini
OPENAI_BASE_URL=
MAX_BATCH_CHARS=8000
MAX_BATCH_BLOCKS=30
MAX_FILE_SIZE_MB=25
```

Экономия токенов:

- Использовать недорогую модель для MVP.
- Включить кэш переводов до вызова API.
- Не делать повторный запрос, если ошибка связана с валидацией входных данных.
- Ограничить максимальный размер файла и batch.

## 11. Кэширование переводов

MVP-кэш:

- In-memory словарь на время обработки одного документа.
- Ключ: нормализованный исходный текст + языковая пара.
- Значение: переведенный текст.

Нормализация:

- Убрать лишние пробелы по краям.
- Схлопнуть повторяющиеся пробелы.
- Сохранять регистр, если он важен для смысла.

Экономия токенов:

- Повторяющиеся заголовки, элементы таблиц и стандартные фразы переводятся один раз.
- Не использовать внешний Redis в MVP.
- Не кэшировать пустые строки и технические обозначения.

## 12. Сохранение форматирования

Стратегия MVP:

- Не создавать документ с нуля.
- Открыть исходный DOCX.
- Заменять текст внутри существующих `paragraph` и `cell`.
- По возможности сохранять стили абзацев, списки, таблицы, изображения и расположение элементов.

Ограничение:

- При простой замене `paragraph.text` может частично теряться форматирование внутри отдельных run.
- Для MVP допустимо базовое сохранение структуры.
- Улучшение run-level замены вынести в следующий этап.

Экономия токенов:

- Не отправлять описание форматирования в AI.
- Не пытаться восстанавливать форматирование через AI.
- Считать структуру задачей DOCX-обработчика, а не языковой модели.

## 13. Расчет стоимости

MVP-режим:

- До перевода считать примерное количество символов.
- Примерно оценивать токены как `characters / 4`.
- Показывать или возвращать оценку в логах/API.
- Реальный биллинг не реализовывать.

Будущий режим:

- Таблица цен по моделям.
- Предварительное подтверждение стоимости.
- Лимиты на документ, пользователя или проект.

Экономия токенов:

- Перед запуском отбрасывать непереводимые блоки из оценки.
- Показывать пользователю примерную стоимость только по переводимому тексту.
- Не делать отдельный AI-запрос для оценки стоимости.

## 14. Логирование и статусы

Статусы обработки:

- `uploaded`
- `parsing`
- `estimating`
- `translating`
- `building`
- `completed`
- `failed`

Логи MVP:

- Имя файла.
- Размер файла.
- Количество текстовых блоков.
- Количество batch-запросов.
- Примерное количество символов.
- Ошибки API без раскрытия секретов.

Экономия токенов:

- Не логировать полный текст документа.
- Не логировать полные ответы AI.
- Для отладки хранить только `batch_id`, размер batch и короткий фрагмент ошибки.

## 15. Надежность

Необходимые механизмы:

- Проверка расширения `.docx`.
- Проверка размера файла.
- Создание папок `uploads/` и `outputs/` при старте.
- Безопасное имя файла.
- Retry для временных ошибок AI API.
- Сохранение частичного прогресса в памяти во время обработки.
- Корректный ответ пользователю при ошибке.

Экономия токенов:

- При retry отправлять тот же batch, а не увеличенный диагностический prompt.
- Не повторять успешно переведенные batch.
- При ошибке одного batch не начинать весь документ заново, если уже есть промежуточные результаты.

## 16. Безопасность

Требования MVP:

- Не доверять имени загруженного файла.
- Не разрешать пути вида `../`.
- Хранить API key только в `.env`.
- Не возвращать пользователю внутренние stack trace.
- Ограничить размер файла.
- Проверять MIME/расширение.

Экономия токенов:

- Не отправлять секреты и внутренние настройки в AI.
- Не включать содержимое `.env` в логи или prompt.
- Не отправлять в AI системные ошибки приложения.

## 17. Тестирование

Минимальный набор тестов:

- Проверка запрета одинаковых языков.
- Проверка отклонения не-DOCX файла.
- Проверка извлечения абзацев.
- Проверка извлечения текста из таблиц.
- Проверка сборки DOCX после замены текста.
- Проверка кэша повторяющихся строк.
- Проверка segmenter на лимиты batch.

Ручные тесты:

- DOCX с заголовками.
- DOCX со списками.
- DOCX с таблицей.
- DOCX с изображением.
- DOCX с техническими обозначениями.

Экономия токенов:

- В unit-тестах мокать AI-клиент.
- Не использовать реальные API-вызовы в обычном тестовом прогоне.
- Для интеграционного теста использовать маленький документ на 2-3 абзаца.

## 18. Этапы реализации

### Этап 1. Базовый каркас проекта

Задачи:

- Создать структуру папок.
- Добавить `requirements.txt`.
- Добавить `.env.example`.
- Создать `app/main.py`.
- Настроить FastAPI и health endpoint.

Результат:

- Приложение запускается через `uvicorn`.

Экономия токенов:

- Не подключать AI на этом этапе.
- Не создавать UI.
- Не писать лишнюю конфигурацию.

### Этап 2. Загрузка и валидация DOCX

Задачи:

- Реализовать endpoint `POST /translate/`.
- Принимать файл и языки.
- Проверять расширение и размер.
- Сохранять файл в `uploads/`.
- Возвращать ошибку при `source_lang == target_lang`.

Результат:

- API принимает корректный DOCX и отклоняет некорректные запросы.

Экономия токенов:

- Не читать содержимое документа до прохождения базовой валидации.
- Не отправлять файл в AI на этом этапе.

### Этап 3. Парсинг DOCX

Задачи:

- Извлекать абзацы.
- Извлекать текст из таблиц.
- Присваивать каждому блоку `block_id`.
- Пропускать пустые блоки.
- Помечать потенциально непереводимые блоки.

Результат:

- Система получает список переводимых блоков.

Экономия токенов:

- Удалять пустые и технические блоки до сегментации.
- Не дублировать одинаковые строки.

### Этап 4. Сегментация и кэш

Задачи:

- Реализовать batch-разбиение.
- Добавить лимиты `MAX_BATCH_CHARS` и `MAX_BATCH_BLOCKS`.
- Добавить in-memory кэш.
- Подготовить JSON-формат для AI.

Результат:

- Текст готовится к переводу экономными batch-запросами.

Экономия токенов:

- Группировать короткие блоки.
- Не переводить повторяющиеся строки.
- Не превышать лимит batch, чтобы не получить дорогие ошибки.

### Этап 5. AI-перевод

Задачи:

- Реализовать OpenAI-совместимый клиент.
- Настроить модель через `.env`.
- Отправлять batch в JSON.
- Валидировать JSON-ответ.
- Добавить retry и timeout.

Результат:

- Сервис получает переводы по `block_id`.

Экономия токенов:

- Использовать короткий prompt.
- Требовать только JSON без объяснений.
- Не отправлять исходный текст повторно после успешного batch.

### Этап 6. Сборка DOCX

Задачи:

- Открыть исходный DOCX.
- Заменить текст в найденных абзацах и ячейках.
- Сохранить результат в `outputs/`.
- Сформировать имя `originalname_translated_to_en.docx`.

Результат:

- Пользователь получает переведенный DOCX.

Экономия токенов:

- Не использовать AI для восстановления структуры.
- Не создавать документ заново, если можно изменить исходный.

### Этап 7. Логи, ошибки и оценка стоимости

Задачи:

- Добавить статусы обработки.
- Добавить краткие логи.
- Добавить примерную оценку символов и токенов.
- Возвращать понятные ошибки API.

Результат:

- Пользователь понимает, что происходит с документом.

Экономия токенов:

- Оценку делать локально, без AI.
- Не логировать полный текст.
- Не добавлять отдельные AI-запросы для диагностики.

### Этап 8. Тесты и ручная проверка

Задачи:

- Добавить unit-тесты ключевых сервисов.
- Проверить маленькие DOCX-файлы вручную.
- Проверить таблицы, списки и заголовки.
- Проверить ошибку одинаковых языков.

Результат:

- MVP можно запускать и демонстрировать.

Экономия токенов:

- Мокать AI-клиент.
- Не тратить реальные API-токены на автоматические тесты.

## 20. Критерии готовности MVP

MVP считается готовым, если:

- Приложение запускается локально.
- DOCX-файл можно загрузить через API.
- Нельзя выбрать одинаковые языки.
- Текст абзацев переводится.
- Текст таблиц переводится.
- Результат сохраняется как DOCX.
- Базовая структура документа сохраняется.
- Повторяющийся текст не отправляется в AI несколько раз.
- Ошибки API обрабатываются без падения приложения.
- Есть базовые тесты без реальных AI-вызовов.

Экономия токенов:

- Готовность MVP оценивать по рабочему DOCX-потоку, а не по числу поддерживаемых форматов.
- Не расширять функциональность до стабилизации основного сценария.
